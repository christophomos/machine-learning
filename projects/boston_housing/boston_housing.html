Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

Question 3 - Training and Testing¶

  • What is the benefit to splitting a dataset into some ratio of training and testing subsets for a learning algorithm?

Hint: Think about how overfitting or underfitting is contingent upon how splits on data is done.

Answer:

Splitting data into testing and training data is most important for avoiding overfitting. You can catch underfitting since models with high bias perform poorly on training as well as testing sets. But a model with high variation may perform excellently on the data it was trained on - it is critical to have a testing set to ensure that the model hasn't simply memorized the training data; that would be overfitting.

But it is critical to keep in mind that splitting data isn't the only answer for how we can avoid underfitting or overfitting. We need a set of data that is representative of the data that we will be applying our model to in the future. For instance, no matter how much 1978 housing data we have, we might immediately suspect that it doesn't apply to the current market, now 40 years later.

And exactly how the splitting is done is critical. We must take a random sampling of the data so that both the training and testing sets are equally representative of the data, and we must not peek at the testing data - it should only be used to evaluate our final model.

Question 7 - Grid Search¶

  • What is the grid search technique?
  • How it can be applied to optimize a learning algorithm?

Hint: When explaining the Grid Search technique, be sure to touch upon why it is used, what the 'grid' entails and what the end goal of this method is. To solidify your answer, you can also give an example of a parameter in a model that can be optimized using this approach.

Answer:

We use grid search to select between different choices of hyperparameters. This is something that training a model won't do for us; training the model only gives us a resulting set of parameters.

For instance, with linear regression, training a model will give us a resulting set of coefficients for our polynomial (parameters), but it will not tell us the degree of the polynomial (a hyper-parameter). So we use grid search to try out all the combinations of hyper-parameters that we're interested in, then use our cross-validation data to choose which combination is best.

So each cell in our grid corresponds to a degree of polynomial, such as 1, 2 or 3. When we execute the grid search, we'll get a model corresponding to each cell in the grid. We then select the best model using the cross-validation data, attempting to avoid high bias (in this case, too low of a degree) and high variation (too high of a degree).

Question 8 - Cross-Validation¶

  • What is the k-fold cross-validation training technique?

  • What benefit does this technique provide for grid search when optimizing a model?

Hint: When explaining the k-fold cross validation technique, be sure to touch upon what 'k' is, how the dataset is split into different parts for training and testing and the number of times it is run based on the 'k' value.

When thinking about how k-fold cross validation helps grid search, think about the main drawbacks of grid search which are hinged upon using a particular subset of data for training or testing and how k-fold cv could help alleviate that. You can refer to the docs for your answer.

Answer:

K-fold cross-validation lets us get maximum use out of a limited set of training data. We are able to avoid the trap of using testing data to choose between models, while at the same time not throwing away a further portion of data that we would use only for cross-validation and not training our model. And the smaller set of training data we're left with, the more susceptible we would be to random choice not sufficiently shuffling our data.

To do this, we split the data into k groups, 1 for each model we wish to evaluate, corresponding to one cell of the grid search grid (1 combination of the hyper-parameters we're choosing between). Each group has a 1/k fraction of our testing set. For each model we want to evaluate, we use one of the groups for cross-validation and the rest of the data for training. The key is to make sure that each cross-validation group is strictly segregated for each model.

from sklearn.tree import DecisionTreeRegressor
from sklearn.metrics import make_scorer
from sklearn.model_selection import GridSearchCV

def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """
    
    # Create cross-validation sets from the training data
    # sklearn version 0.18: ShuffleSplit(n_splits=10, test_size=0.1, train_size=None, random_state=None)
    # sklearn versiin 0.17: ShuffleSplit(n, n_iter=10, test_size=0.1, train_size=None, random_state=None)
    #cv_sets = ShuffleSplit(X.shape[0], n_iter = 10, test_size = 0.20, random_state = 0)
    cv_sets = ShuffleSplit(n_splits=10, test_size=0.2,random_state = 0)

    # Create a decision tree regressor object
    regressor = DecisionTreeRegressor()

    # Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    params = {'max_depth': list(range(1,11))}
    #print(params)

    # Transform 'performance_metric' into a scoring function using 'make_scorer' 
    scoring_fnc = make_scorer(performance_metric)

    # Create the grid search cv object --> GridSearchCV()
    # Make sure to include the right parameters in the object:
    # (estimator, param_grid, scoring, cv) which have values 'regressor', 'params', 'scoring_fnc', and 'cv_sets' respectively.
    grid = GridSearchCV(regressor, params, scoring_fnc, cv=cv_sets)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)

    # Return the optimal model after fitting the data
    return grid.best_estimator_

Answer:

  • 1978 is simply too long ago. 1978 is a lifetime of real estate trends removed. Here's an example of some major trends from https://boston.curbed.com/2018/5/30/17404868/boston-housing-why-high:

    Its not just population growth in the Boston region. Its that many of these new residents make the kind of coinage that allows them to afford 700,000 one-bedrooms and 2.5 million colonials. Sometimes being one of the worlds economic engines has its drawbacks.

  • There aren't enough features here. The "Zestimate" is the gold standard of online housing estimates (noting of course that an actual appraisal is the gold standard of appraisals), and here is what their web site says on the subject:

    Specifically, some of the data we use in this algorithm include: Physical attributes: Location, lot size, square footage, number of bedrooms and bathrooms and many other details. Tax assessments: Property tax information, actual property taxes paid, exceptions to tax assessments and other information provided in the tax assessors' records. Prior and current transactions: Actual sale prices over time of the home itself and comparable recent sales of nearby homes"

  • The range of over 69 thousand dollars in the model above is too high. This points to a lack of robustness.
  • The data collected in an urban city may be relevant in a rural area, but it would be better to do research in every area and specifically investigate which data are relevant across areas and which have more or less predictive value in specific settings or geographic regions.
  • Is it fair to judge a house by its neighborhood? The question is, if you're looking to sell your house, you have to face the reality that certain neighborhoods command higher premiums. And those are for reasons that can't be brushed aside, such as quality of public schools, crime, and access to transportation. But of course, things like racism can also be at work, and as machine learning practicioners, we should seek to avoid encoding human prejudices in our models.
